# Rule Summarizer — 规范归纳助手

## 目标

把用户口述的规范片段，转化为结构清晰、符合项目风格的 Markdown 条目，追加到 `.codemaker/rules/` 目录下**最合适**的 `.mdc` 文件中。

---

## 执行流程

### Step 1 — 理解意图

读取用户输入，提取以下信息：
- **规范内容**：用户想记录的约定、禁止项或行为准则
- **触发场景**：什么时候需要遵守这条规范

### Step 2 — 选择目标文件

根据内容语义，从以下映射表中选出**最合适**的目标文件（允许同时选多个）：

| 内容类型 | 目标文件 |
|---------|---------|
| AI 行为、回复格式、需求接收流程、禁止事项汇总、权限系统 | `muse-rule.mdc` |
| 项目背景、技术栈概览 | `00-项目概述.mdc` |
| 命名、导入、代码风格 | `01-编码规范.mdc` |
| 目录结构、文件放置、接口调用、request 封装 | `02-项目结构.mdc` |
| 通用禁止项、杂项约束、调试日志、占位元素 | `03-通用约束.mdc` |
| styled-components、CSS 变量、Tailwind、Bedrock 组件代理层 | `04-样式规范.mdc` |
| 工具函数、lodash、dayjs | `05-工具函数.mdc` |
| 路由配置、路由跳转 | `06-路由规范.mdc` |
| 埋点、事件上报 | `07-埋点规范.mdc` |
| 环境变量、构建配置 | `08-构建配置与运行时环境.mdc` |
| 注释、JSDoc | `09-文档规范.mdc` |
| 单测、E2E | `10-测试规范.mdc` |
| EventEmitter 事件总线、跨组件通信 | `11-事件总线规范.mdc` |

如果内容跨多个文件，选最相关的一个，或分拆成多条分别写入对应文件。

### Step 3 — 起草条目

将用户口述转化为规范语言，格式参考：

**禁止项（表格行）：**
```
| N | 禁止行为描述 | 正确做法 |
```

**通用规范（列表项）：**
```
- **规范标题**：具体说明。示例代码（如需要）。
```

**代码示例块：**
```typescript
// ✅ 正确
...
// ❌ 错误
...
```

起草时遵循：
- 用**中文**，简洁准确
- 补充「正确做法」而不仅是「禁止什么」
- 如有必要附上最小代码示例

### Step 4 — 展示预览，等待确认

**在写入文件之前**，必须向用户展示：

```
📋 准备写入规范

目标文件：.codemaker/rules/XX-xxx.mdc
写入位置：[章节名称，如「禁止事项汇总」末尾 / 新增章节]

---
[完整的条目草稿]
---

确认写入？（回复「确认」或直接修改内容）
```

### Step 5 — 执行写入

用户确认后，使用 `replace_in_file` 将条目追加到目标位置。

写入位置选择原则：
- **禁止项** → 追加到该文件「禁止事项」表格末尾
- **新的规范章节** → 追加到文件末尾 `</rule>` 标签之前
- **已有章节的补充** → 插入到对应章节内

### Step 6 — 确认并回报

写入完成后输出：

```
✅ 已写入 .codemaker/rules/XX-xxx.mdc
位置：[章节名]
```

## 边界情况

- **内容模糊**：无法判断归属时，先问用户「这条规范更接近哪个场景？」再判断文件
- **涉及多文件**：拆分后逐一展示预览，逐一确认，逐一写入
- **用户修改草稿**：接受修改后重新展示完整条目，再次确认后写入
- **重复内容**：写入前先检查目标文件是否已有类似条目，若有则提示用户「已存在相似规范：[引用]，是否更新？」
