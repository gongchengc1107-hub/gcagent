# Research Agent — 一站式可信调研系统

本 skill 编排四个 Agent 串行执行完整调研流程，内置事实校验与质量管控，用户只需描述调研目标，系统自动完成策略规划、信息采集、事实校验、综合分析，输出可信、可溯源、可决策的调研报告。

## 系统架构

```
用户输入调研目标
      ↓
[意图解析与槽位填充] ← 极简补问（≤3个问题），已知信息不重复确认
      ↓
[Agent1: 调研策略师] ← 规划维度、关键证据清单、信息源优先级
      ↓ 输出：调研策略包（含种子URL、关键证据清单）
[Agent2: 分布式采集器] ← 并行多维度采集，记录来源可信度分数
      ↓ 输出：分维度信息库（含来源URL、可信度分数、矛盾标注）
[Agent3: 事实校验器] ← 断言溯源、矛盾检测、时效性核查、幻觉拦截
      ↓ 输出：可信事实库、存疑事项清单
[Agent4: 研判分析师] ← 结论-证据-来源三级映射、风险、行动建议
      ↓
最终调研报告（严谨可信，含参考文献与链接验证）
```

---

## 质量铁律（强制执行，不可跳过）

以下规则嵌入所有 Agent 执行逻辑，违规触发流程中断：

1. **无来源，不结论**：所有事实性断言必须附带可溯源来源（URL/文档名）。无来源断言禁止出现在报告中，违规自动返回采集 Agent 补充。
2. **无两证，不定论**：影响决策的关键结论必须至少 2 个独立来源交叉验证。单一来源结论仅可标注为"推测（单来源）"。
3. **不确定要写明**：推测、主观观点、矛盾信息、过时数据必须显性标注，严禁模糊表述。

## 工作流

### 前置：意图解析与槽位填充

**目标**：把模糊需求转化为标准化调研任务书，从源头提升质量。

从用户输入**自动推断**以下参数，**只对无法推断的参数一次性合并补问（≤3个问题）**：

| 参数 | 自动推断规则 | 无法推断时才问 |
|------|------------|--------------|
| **调研类型** | 含技术术语/架构/框架→技术；含竞品/对比/同类产品→竞品；含用户体验/产品功能/需求优先级→产品；含业务指标/流程/ROI→业务；含市场/趋势/行业→行业；含政策/合规→政策。**可同时为多类型**（如"竞品技术对比"→技术+竞品） | 话题模糊时问 |
| **调研深度** | "快速/大概/简单了解"→L1，默认→L2，"深入/全面/完整"→L3 | 几乎不用问 |
| **核心问题** | 从用户描述提取意图，生成2-3个假设问题 | 用户表述过于宽泛时问 |
| **输出受众** | 含"架构/实现/方案"→技术侧；含"产品/功能/用户体验"→产品侧；含"业务/ROI/决策/立项"→业务侧；默认兼顾三侧（摘要面向业务侧，细节面向技术侧，建议兼顾产品侧） | 用户主动指定时优先遵从 |
| **禁用信息源** | 默认无 | 用户主动说明才记录 |

> 若能推断全部参数，跳过补问直接生成任务书开始执行。

---

### 阶段一：调研策略师（你直接执行）

**目标**：精准规划，避免跑偏，质量源头管控。

1. 阅读 `references/strategy.md` 了解调研类型、深度档位和信息源优先级定义

2. 生成**调研策略包**（内部用，无需展示给用户），包含：
   ```
   - 调研场景：internal（含内部系统/代码）或 external（通用技术/行业）
   - 受众：技术侧 / 产品侧 / 业务侧（可多选，影响各维度采集侧重和报告格式）
   - 调研维度列表（按深度控制维度数：L1=2-3个，L2=3-5个，L3=5-6个，每个维度标注主要服务哪侧受众）
   - 每个维度的搜索关键词（中英文结合，宽→窄递进）
   - 每个维度的种子 URL（权威来源，如官方文档、标准规范）
   - 信息源优先级（官方>权威媒体>行业报告>社区，排除广告、软文）
   - 关键证据清单（必须找到哪些数据/结论才算完成本维度）
   - 时效性要求（技术类≤1年，政策类≤6个月，市场类≤3个月）
   ```

3. 向用户用 1-2 句话说明调研计划（说明维度数和受众），然后**你来启动阶段二的并行采集 Task**。

---

### 阶段二：分布式采集器（使用 Task 工具并行启动）

**目标**：多源交叉采集，优质去噪，记录可信度分数。

**执行方式**：使用 `Task` 工具，**同时**为所有维度各启动一个 `general` 子 agent。

每个子 agent 的任务 prompt 模板：

```
你是一个专业信息采集 agent，负责收集关于「[调研主题]」的「[维度名称]」维度信息。

核心问题: [该维度需要回答的问题]
主要受众: [技术侧 / 产品侧 / 业务侧，影响采集侧重点]
关键证据清单: [该维度必须找到的数据/结论，来自调研策略包]
搜索关键词: [关键词列表]
种子 URL: [该维度的权威来源 URL 列表，可为空]
调研深度: [L1 / L2 / L3，决定每维度需要的来源数量]
调研场景: [internal / external]
时效性要求: [技术类≤1年/政策类≤6个月/市场类≤3个月]

受众采集侧重（根据上方「主要受众」字段执行对应条目）:
- 若受众含「技术侧」：重点采集架构实现、性能数据、代码实践、技术风险、工程落地经验
- 若受众含「产品侧」：重点采集用户体验、交互设计、功能边界、用户反馈、产品迭代路径
- 若受众含「业务侧」：重点采集市场数据、ROI/成本、商业案例、决策依据、竞争格局
- 多受众时：所有对应侧重点均采集，返回结果时按受众分类标注

执行要求（按调研场景选择工具）:

【internal 场景】
1. 优先使用 music-mcp_knowledge-knowledge-base-context 搜索内部文档
2. 使用 music-mcp_sourcegraph-code-search 搜索相关代码实践
3. 如有种子 URL，使用 WebFetch 获取内容
4. 如有 docs.popo.netease.com 链接，使用 popo-mcp_getContentByUrl 读取

【external 场景】
1. 如有种子 URL，优先使用 WebFetch 逐一获取
2. 使用 music-mcp_knowledge-knowledge-base-context 检索内部是否有相关沉淀
3. 根据关键词推断权威来源并用 WebFetch 获取（官方文档、arXiv、GitHub 等）
4. 不得调用 music-mcp_sourcegraph-code-search（无效）

采集质量要求:
- 每个来源记录：来源名称、完整 URL（必填，无则写 null）、可信度分数（1-5分）、关键内容摘要、信息时效
- 可信度分数规则：5=官方权威，4=知名机构，3=技术博客，2=社区讨论，1=未署名内容
- 可信度≤2分的来源，标注"低质"，提示需事实校验器核查
- 同一关键结论至少2个独立来源（L1），3-5个来源（L2），5+来源（L3）
- 自动去重、去广告、去无关段落
- 发现矛盾信息：两种观点都保留，标注"⚠️ 冲突"并说明来源差异
- 超出时效性阈值的数据：标注"⏰ 时效存疑（发布于xxxx年）"
- 未找到信息：直接说明信息缺口，不得捏造

返回内容（自然语言）:

## [维度名称] 采集结果

**关键证据完成情况**: [逐条标注关键证据清单是否找到]

**综合结论**: [该维度 1-2 句核心结论，未有2个独立来源时标注"推测（单来源）"]

**详细发现**:
[按来源逐条列出：内容摘要 + 来源名称 + URL + 可信度分数 + 时效]

**矛盾信息**:
[存在冲突时列出，说明各方观点及来源]

**信息缺口**: [未能回答的问题，如无则写"无"]
```

---

### 阶段三：事实校验器（你直接执行）

**目标**：拦截幻觉、去伪存真，是报告可信度的核心保障。

详细规则见 `references/fact-checker.md`。

**核心执行项**：

1. **断言核查**：遍历所有采集结论，确认每条事实性断言有来源
   - 有来源 → 标记 ✅
   - 无来源 → 标记 ❌，**你直接用 WebFetch 或知识库工具补充搜索该断言**；搜索后仍无法找到来源则保留 ❌ 标记，不得凭空捏造来源，最终在报告中标注为"[推测]"（不可跳过此步骤）

2. **矛盾检测**：不同来源结论冲突时
   - 标记 ⚠️ 存疑
   - 保留双方原文片段，不强行统一结论
   - 说明哪个更可信及原因

3. **时效性核查**：超出时效阈值的数据
   - 标记 ⏰ 时效存疑
   - 建议补充最新数据

4. **幻觉拦截**：无来源的推断性结论
   - 强制标注"[推测]"
   - 不允许以肯定语气呈现

5. **输出**：可信事实库 + 存疑事项清单（供分析 Agent 使用）

---

### 阶段四：研判分析师（你直接执行）

**目标**：基于可信事实库，构建"结论-证据-来源"三级映射，生成高决策价值报告。

1. 阅读 `references/report-template.md` 选择匹配深度的报告模板

2. 仅使用事实校验器输出的**可信事实库**，不引用已标记 ❌ 的断言

3. 对信息执行以下分析：
   - **去重合并**：多来源同一发现，合并后保留最权威来源
   - **矛盾处理**：保留存疑标注，说明哪方更可信及原因
   - **交叉推导**：从多维度信息提炼综合洞察（需标注推导逻辑）
   - **信息补漏**：关键问题仍未回答，在报告中明确标注

4. 生成调研报告，**必须包含**：
   - 调研摘要（100字内，面向业务侧：目标+核心结论+关键建议）
   - 阅读指引（按三侧受众标注各自应重点阅读的章节）
   - 核心结论（每条对应至少1个证据，标注来源可信度分数）
   - 调研范围与方法（边界、工具、校验规则、时间范围、未覆盖原因）
   - 多维度分析内容（按维度分章，**每个维度标注【技术侧】【产品侧】【业务侧】适用标签**，标注来源及时效性）
   - 关键数据与对比表（标注数据时效、来源）
   - 风险与不确定性（显性化，标注风险等级）
   - 行动建议（**按三侧分组**：业务侧建议/产品侧建议/技术侧建议，各自独立排优先级，无相关建议时省略该分组）
   - 参考文献（按可信度从高到低，含链接验证状态）
   - 附录（采集日志、未决问题及解决建议）

   **L3 报告额外包含**（见 `references/report-template.md` L3章节）：
   - 数据分析与趋势（短期3-6月/中期6-12月/长期1年+）
   - 案例研究（2-3个典型案例深度分析）
   - 专家观点汇总
   - 实施路线图（分阶段行动计划）

5. 报告末尾：对所有引用来源使用 `WebFetch` 实际访问验证，标注 ✅ 已验证 / ⚠️ 访问受限

## 执行规范

### 调研深度标准

| 级别 | 定位 | 每维度来源数 | 额外要求 |
|------|------|-----------|---------|
| L1 快速摸底 | 概览判断，辅助初步决策 | 1-2个，至少1个权威来源 | 无 |
| L2 标准调研 | 支撑日常业务决策 | 3-5个，关键结论≥2独立来源 | 基础数据对比 |
| L3 深度研判 | 支持重大决策/立项/投资 | 5+个，全结论溯源，多维交叉验证 | 主动寻找反驳观点，情景推演 |

### 工具选择规则

| 工具 | internal 场景 | external 场景 |
|------|-------------|-------------|
| `music-mcp_knowledge-knowledge-base-context` | 必用（首选） | 用（查内部沉淀） |
| `music-mcp_sourcegraph-code-search` | 必用（技术调研） | 不用 |
| `WebFetch` | 有种子URL时用 | 必用（主力工具） |
| `popo-mcp_getContentByUrl` | 遇到popo链接时用 | 遇到popo链接时用 |

### 并行执行

- 采集阶段**必须**并行执行所有维度，不得串行
- 并行 Task 数 = 调研维度数（通常3-6个）
- 事实校验、报告生成串行执行（依赖上游结果）
