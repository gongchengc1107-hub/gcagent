# 组件一致性规则 + 变更规范

> Phase 4-N 绘制过程中持续参考，确保跨页面组件视觉完全一致。

---

## 一、组件一致性核心原则

### 1.1 同类组件同一实例（NON-NEGOTIABLE）

**规则**：相同用途的组件，在所有页面中必须使用同一个 reusable 组件实例。

```
✅ 正确：
  - 供应商列表页的「状态标签」和作品列表页的「状态标签」→ 使用同一个 StatusTag 组件
  - 所有弹窗的「取消按钮」→ 使用同一个 Button-secondary 组件实例

❌ 错误：
  - 同一类型的按钮，在不同页面使用了颜色略有不同的两个版本
  - 卡片组件在 A 页面圆角 8px，在 B 页面圆角 6px
```

**执行方式**：在 Phase 4 每批次绘制前，先查询 Phase 2 的「组件使用规划表」，确认使用哪个 reusable ID，再通过 `pencil_batch_design` 的 `ref` 引用插入。

### 1.2 组件属性覆盖规则

当需要对 reusable 组件的实例做局部定制时：

```
✅ 允许的实例级覆盖（不修改 reusable 本身）：
  - 更新实例内的文字内容（content）
  - 切换 reusable 组件预设的变体（variant）
  - 调整实例的 width / height（仅当组件支持弹性尺寸时）

❌ 不允许的操作（会影响所有使用该组件的地方）：
  - 直接修改 reusable 组件本身的颜色、间距、字号
  - 在组件内部增删子节点（除非明确需要"变更组件"）
```

### 1.3 新建组件的条件

只有满足以下**任一**条件时，才允许新建 reusable 组件：

| 条件 | 说明 |
|------|------|
| 现有组件无法覆盖 | .pen 文件和 Bedrock 均无对应组件 |
| 需要多状态变体 | 业务状态超过 2 种，需要统一管理样式 |
| 复用频率高 | 该组件将在 3 个或以上页面出现 |
| 样式差异显著 | 与现有组件差异超过 30%，改造成本高于新建 |

## 二、组件分类与复用优先级

### 2.1 复用决策树

```
需要一个组件 →

① 查 Phase 2 组件规划表，是否已有对应 .pen reusable？
   ├─ 是 �� 直接用，不修改原组件
   └─ 否 → 继续 ②

② 查 Bedrock 组件库，是否有标准组件满足需求？
   ├─ 是 → 使用 Bedrock 组件（bedrock-component-lookup 确认 API）
   └─ 否 → 继续 ③

③ 现有 .pen reusable 是否可以通过"变更"来满足？
   ├─ 是 → 变更现有组件（必须输出变更说明，等用户确认）
   └─ 否 → 新建 reusable 组件（必须输出新增说明，等用户确认）
```

### 2.2 常见组件复用规则

| 组件类型 | 优先复用方式 | 注意事项 |
|---------|------------|---------|
| **按钮** | Bedrock Button（primary/secondary/danger）| 不要自绘按钮 |
| **输入框** | Bedrock Input / Select / DatePicker | 保持统一高度（32px）|
| **弹窗** | 已有 .pen Modal reusable 或 Bedrock Modal | 内容区可替换，框架不改 |
| **状态标签** | 已有 .pen StatusTag 或新建 | 各业务状态颜色必须统一 |
| **表格** | Bedrock Table | 列宽、行高保持项目统一规范 |
| **表单** | Bedrock Form + FormItem | 标签宽度全局统一（如 80px）|
| **空状态** | 已有 .pen EmptyState 或新建 | 图标/文案可替换 |
| **分页** | Bedrock Pagination | 位置统一（表格底部右对齐）|
| **顶部导航** | 已有 .pen Header/NavBar reusable | 禁止在不同页面用不同版本 |
| **面包屑** | Bedrock Breadcrumb | 层级超过 2 级才显示 |

## 三、组件变更规范

### 3.1 变更类型定义

| 类型 | 定义 | 影响范围 |
|------|------|---------|
| **新增** | 创建全新的 reusable 组件 | 无（新建） |
| **变更** | 修改 reusable 的结构/层级/插槽 | 所有引用该组件的实例 |
| **优化** | 调整视觉细节（间距/字号/圆角）不改结构 | 所有引用该组件的实例 |
| **修改** | 更改颜色、状态逻辑等属性 | 所有引用该组件的实例 |

> ⚠️ **变更/优化/修改** 均会影响所有使用该组件的页面，必须告知用户影响范围。

### 3.2 变更说明输出格式

```
📦 组件变更说明

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[新增] {组件名称}
  触发原因：{需求中出现了哪个场景，现有组件无法满足}
  组件用途：{该组件将用于哪些页面/场景}
  包含变体：{列出状态变体，如：待审核/已通过/已拒绝}
  影响页面：无（新建）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[变更] {组件名称} (ID: {layerId})
  修改内容：{具体修改了什么，如"在标题区新增了操作按钮插槽"}
  修改原因：{为什么需要这个修改，现有结构无法满足的场景}
  影响页面：{列出所有已引用该组件的页面，如"供应商列表页、作品列表页"}
  是否向后兼容：{是（原有实例不受影响）/ 否（原有实例需要检查）}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[优化] {组件名称} (ID: {layerId})
  调整内容：{如"内边距从 8px 调整为 12px"}
  优化理由：{如"当前页面内容较多，需要更宽松的布局以提升可读性"}
  影响页面：{列出所有已引用该组件的页面}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 3.3 用户确认流程

```
输出变更说明后：

用户回复"同意" 或 继续输入（默认同意）
  → 记录变更已确认
  → 继续绘制

用户回复"调整 {具体意见}"
  → 按意见修改组件
  → 截图确认效果
  → 重新输出变更说明
  → 再次等待确认

用户回复"恢复原样" 或 "不改"
  → 撤销当前变更
  → 回到复用决策树，寻找替代方案
  → 若无替代方案，向用户说明情况并讨论解决思路
```

## 四、视觉一致性检查清单

每批次绘制完成后，截图前必须逐项检查：

### 4.1 颜色一致性

- [ ] 所有主操作按钮颜色一致（使用 `--primary-1`）
- [ ] 所有次操作按钮颜色一致（使用 `--fill-3` 背景 + `--text-color-1` 文字）
- [ ] 相同业务状态的标签颜色完全一致（如"待审核"始终是警告黄）
- [ ] 页面背景色一致（使用 `--bg-1`）
- [ ] 卡片背景色一致（使用 `--bg-2`）
- [ ] 无硬编码颜色（无 `#XXXXXX` 格式的颜色值）

### 4.2 间距一致性

- [ ] 相同类型的组件内边距一致（基于 4px 栅格）
- [ ] 列表项间距一致
- [ ] 页面内容区边距一致（通常 24px 或 32px）
- [ ] 表单标签与输入框的间距一致

### 4.3 字号一致性

- [ ] 页面标题字号一致（`text-xxl` 20px）
- [ ] 卡片/模块标题字号一致（`text-lg` 16px）
- [ ] 正文字号一致（`text-base` 14px）
- [ ] 辅助文字字号一致（`text-sm` 12px）

### 4.4 组件实例一致性

- [ ] 同类按钮使用同一个 reusable 实例
- [ ] 同类标签使用同一个 reusable 实例
- [ ] 弹窗框架使用同一个 reusable 实例
- [ ] 表格列宽规则一致

### 4.5 状态完整性

- [ ] 对照 `interaction-logic.md`，所有定义的状态均已绘制
- [ ] 每个状态帧的帧标题格式正确（`{页面名} — {状态名}`）
- [ ] 未出现文档中未定义的状态帧
- [ ] 空状态已绘制（所有列表/数据展示页）

## 五、禁止行为（NON-NEGOTIABLE）

```
❌ 禁止在不知情的情况下静默修改已有 reusable 组件
❌ 禁止在未告知原因前继续绘制下一页
❌ 禁止将"优化"包装成不需要说明的微小调整
❌ 禁止跨页面使用视觉不一致的同类组件
❌ 禁止使用硬编码颜色值（#XXXXXX）
❌ 禁止绘制 interaction-logic.md 中未定义的状态
❌ 禁止遗漏 interaction-logic.md 中已定义的状态
```
