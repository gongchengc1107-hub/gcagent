# Skill: openspec-implement-change

## 描述

驱动 IMPLEMENT 阶段的代码实现，每完成一个 task 立即在 `tasks.md` 中勾选 `[x]`，
保持任务进度与代码实现的**实时同步**。

目标：**零歧义的任务驱动实现**——不猜测需求，不跳过任务，不合并勾选。

---

## 触发时机

- 用户说"开始实现"、"执行 IMPLEMENT"、"按 tasks 写代码" 时
- REVIEW / API-AUDIT 阶段已通过（`review.md` 结论为 APPROVED）

---

## 前置检查（Start Guard）

执行前必须验证以下条件，**任意一条不满足则停止并提示**：

1. `openspec/changes/{change-id}/tasks.md` 存在且非空
2. `openspec/changes/{change-id}/design.md` 存在
3. `openspec/changes/{change-id}/review.md` 存在，且结论为 `APPROVED`
4. `openspec/changes/{change-id}/api-audit.md` 存在（若 design.md 中引用了任何 API）

> 若 api-audit.md 中存在 `BLOCKED` 状态的接口，停止并告知用户哪些接口缺失，
> 并建议先联系后端补充或调整方案。

---

## 执行步骤

### Step 1：读取任务列表

读取 `tasks.md`，解析所有任务条目：
- `- [ ]` → 待实现
- `- [x]` → 已完成，跳过

从**第一个 `- [ ]` 任务**开始执行。

---

### Step 2：逐任务实现（Task Loop）

每个任务按以下流程处理：

#### 2.1 实现前声明
在回复中用以下格式声明当前任务：
```
▶ 正在实现：{任务分组} > {任务名称}
```

#### 2.2 执行实现
- 严格依据 `design.md` 中的 Decisions 实现，不自行发散
- 遵循 `openspec/config.yaml` 中的技术规范约束
- **禁止将代码输出到会话消息中**，所有代码必须通过 `edit_file` / `replace_in_file` 直接写入文件
- 若 `design.md` 或 `review.md` 中包含 MasterGo 链接，在实现 UI 相关任务前使用 `mcp__getDsl` 读取设计稿，以设计稿为 UI 实现的最终标准（间距、颜色 token、组件结构等）
- 若遇到 design.md 未说明的细节，**优先查阅现有代码风格**，再小范围决策
- 若决策影响超出任务范围，**暂停并询问用户**，不静默扩展

#### 2.2.1 UI 任务完成后自动触发设计验收
当完成的任务属于 UI 相关（页面骨架、样式打磨、组件实现等），自动触发 `design-qa-workflow` 阶段 1：
1. 检查 `design-specs/{需求名称}/_manifest.json` 是否存在
2. 存在 → 根据 `qaLevel` 执行 A 级或 B 级验收
3. 不存在 → 仅做基本 CSS 合理性检查，不阻塞
4. 验收未通过项 → 立即修复后重新验收，确认通过后才进入下一步勾选

#### 2.3 实现完成后执行自查 → 勾选

任务对应代码写入完毕后，**必须立即执行自查清单**（`00-自查清单.mdc` 全部 10 项，必须实际运行工具验证，不允许凭记忆跳过）：

- **自查通过** → 在 `tasks.md` 中将该条目从 `- [ ]` 改为 `- [x]`
- **自查未通过**（存在 ❌ 项）→ 必须先修复所有 ❌ 项，再次执行自查，通过后才可勾选
- **禁止在自查未通过时勾选任务或进入下一个任务**

```markdown
// tasks.md 中
- [x] 配置路由 & access 权限  ← 自查通过，立即勾选
- [ ] 实现列表页面骨架
```

> ⚠️ 禁止"批量勾选"——必须一个任务完成一个勾选，保持进度真实可查。

#### 2.4 自动 Git Commit（易协作单模式）

**当需求分析阶段用户提供了易协作单（`ticketRef`）时**，每个任务完成并勾选后，必须先触发 `code-simplifier` Skill 简化代码，再执行 `git commit`：

```bash
git add -A && git commit -m "feat: refs {ticketRef}" -m "{任务名称}"
```

**示例：**
```bash
# 用户提供的易协作单为：#12345 供应商模块优化
git add -A && git commit -m "feat: refs #12345 供应商模块优化" -m "配置路由 & access 权限"
```

规则：
- `git add -A` 提交所有文件变更，包括 `openspec/`、`.codemaker/rules/`、`.codemaker/skills/` 等控制流程文件
- 第一行 `-m` 为 `feat: refs {ticketRef}`（ticketRef 即用户提供的 `#单号 单名` 原文）
- 第二行 `-m` 为当前完成的任务计划小点名称
- **没有易协作单时跳过此步骤**，行为与之前一致
- 若 `git commit` 失败（如没有文件变更），记录警告但不阻塞后续任务

#### 2.5 完成汇报
每个任务完成后输出一行：
```
✅ 完成：{任务名称}（共 {已完成数}/{总数} 个任务）
```

---

### Step 3：分组完成提示

每个分组（如「权限/路由」「API」「页面骨架」）的最后一个任务完成后，输出：
```
📦 分组「{分组名}」已完成，继续下一分组？
```

**等待用户确认**后再继续，给用户检查代码的机会。

---

### Step 4：所有任务完成

全部 `- [ ]` 任务勾选完毕后，输出完成摘要：

```markdown
## 🎉 IMPLEMENT 完成

**变更 ID**: {change-id}
**完成时间**: {datetime}

### 任务完成情况
| 分组 | 完成数 | 总数 | 状态 |
|------|--------|------|------|
| 权限/路由 | n | n | ✅ |
| API | n | n | ✅ |
| 页面骨架 | n | n | ✅ |
| 交互逻辑 | n | n | ✅ |
| 样式打磨 | n | n | ✅ |
| 测试与验收 | n | n | ✅ |

### 下一步
执行 POST-REVIEW 阶段 → 运行 `openspec-post-review-change`
```

## 异常处理

### 遇到任务依赖关系冲突
- 描述冲突：「任务 A 依赖 B 的产出，但 B 尚未完成」
- 询问用户：调整顺序 or 手动处理

### 遇到 API 不存在
- **立即停止该任务**，不猜测入参/出参
- 输出：`⛔ 接口 {functionName} 在 src/OpenApiServices/ 中不存在，请先执行 pnpm api:genapi 或确认 API-AUDIT 结果`

### 遇到 design.md 信息不足
- 不猜测，直接输出：`⚠️ 任务「{任务名}」需要更多信息：{具体缺失内容}`
- 等待用户补充后继续

---

## 注意事项

- **禁止**在实现完成前勾选任务（防止进度失真）
- **禁止**跳过任务直接实现后续任务
- **禁止**在实现时引入 design.md 未提到的新依赖
- 每次会话开始时重新读取 `tasks.md` 最新内容，防止缓存导致进度丢失
