# CodeMaker 预设指令使用说明

---

## 一、Prompts（通用指令）

### @useRule 窗口新开时调用

-   作用： 可以保证后续聊天都按规范来
-   聊天内容：
    @/.codemaker/prompts/useRule.md

### @genRule 生成项目rule

-   作用：完整按照模板去分析仓库然后生成一般初始化rule文件
-   聊天内容：
    @/.codemaker/prompts/genRule.md， 【还有什么要注意的编码方面的可以添加上去】

### @adjustRule 转换旧项目rule为新的按照模板来的rule

-   作用： 调整项目已有rule文件，让它按照模板来革新，质量高一些，rule内容不会进行大调整
-   聊天内容：
    @/.codemaker/prompts/adjustRule.md 待调整rule: 【@自己的rule文件】

### @addRule 添加项目rule

-   作用： 发现AI产出不好的代码时，可以添加新的rule
-   聊天内容：
    @/.codemaker/prompts/addrule.md rule描述：【简要描述自己想添加的约束】
-   【可选：可以把仓库里面的好代码贴上去以及把AI产出的坏代码贴上去】

### @updateRule 更新项目rule

-   作用： 发现rule里面哪些还缺了示例，或者示例不是很好时可以删掉示例，然后重新调整更新
-   聊天内容：
    【选中要更新的rule后】@/.codemaker/prompts/updateRule.md 【可选：附带好坏代码示例】

### @ask 让提问更高质量

-   作用： 引导AI反向提问后，AI会先向你确认一些问题后再给出方案，产出质量更高
-   聊天内容：@/.codemaker/prompts/ask.md

### @pref 提交前重新设计，调整和及时清理纠正垃圾代码

-   作用： 非常关键的一步，针对当前聊天批次的上下文去综合调整和优化
-   聊天内容：@/.codemaker/prompts/pref.md

## 二、Skills（开发技能指令）

> Skills 是面向具体开发场景的预设指令，可以快速生成符合项目规范的代码骨架。
> 大部分 Skill 会被 AI 自动识别触发，也可以在聊天中手动提及对应关键词。

### 📋 需求与规划类

#### 需求分析工作流

-   作用：处理需求文档、设计稿、接口分析
-   触发方式：说"做需求"、"更新需求"，或提供 POPO 文档链接
-   对应文件：`.codemaker/skills/需求分析工作流/SKILL.md`

#### 大需求拆分实施

-   作用：将大型需求拆分为多个独立 Change，分批实现
-   触发方式：需求预估 > 40h 或涉及 > 3 个页面时自动建议，或说"做大需求"、"拆分需求"
-   对应文件：`.codemaker/skills/large-requirement-workflow/SKILL.md`

### 🧩 组件开发类

#### @createComponent 创建标准组件

-   作用：根据描述自动生成符合项目规范的 React 组件（简单/复杂/完整模块）
-   触发方式：说"创建一个组件"、"新建组件"，或 AI 计划编写新组件时自动触发
-   对应文件：`.codemaker/skills/create-component/SKILL.md`

#### @createPage 创建页面模块

-   作用：创建完整的路由页面模块，包含 index.tsx、usePresenter、样式、子组件目录等
-   触发方式：说"创建一个页面"、"新建页面"
-   对应文件：`.codemaker/skills/create-page/SKILL.md`

#### @createPresenter 为已有组件抽取 Presenter

-   作用：将已有组件中过重的逻辑拆分到 usePresenter.tsx，让视图层保持干净
-   触发方式：说"抽取 presenter"、"提取逻辑"，或组件代码超过 100 行时 AI 自动建议
-   对应文件：`.codemaker/skills/create-presenter/SKILL.md`

### 📡 数据请求类

#### @createRequestHook 创建请求 Hook

-   作用：创建封装了数据请求逻辑的自定义 Hook，支持自动/手动/分页模式
-   触发方式：说"创建请求 hook"、"封装请求逻辑"
-   对应文件：`.codemaker/skills/create-request-hook/SKILL.md`

#### @updateApi 更新接口文件

-   作用：从指定环境（test1/test2/test3/uat）拉取最新接口定义，生成 OpenApiServices
-   触发方式：说"更新接口"、"刷新API"、"同步接口"、"拉取接口"
-   对应文件：`.codemaker/skills/update-api/SKILL.md`

### 🗃️ 状态管理类

#### @createModel 创建 umi Model

-   作用：在 src/models/ 下创建全局状态管理文件，遵循 umi model 模式
-   触发方式：说"创建一个 model"、"新建 model"
-   对应文件：`.codemaker/skills/create-model/SKILL.md`

### 🛠️ 通用工具类

#### @createHook 创建自定义 Hook

-   作用：在 src/hooks/ 下创建可复用的自定义 Hook，优先组合 ahooks 已有功能
-   触发方式：说"创建一个 hook"、"新建 hook"
-   对应文件：`.codemaker/skills/create-hook/SKILL.md`

#### @bedrockLookup 查询 Bedrock 组件

-   作用：查询 @bedrock/components 组件是否存在及其 API 用法，自动缓存到本地
-   触发方式：说"用 Bedrock 的 XXX 组件"、"Bedrock 的 XXX 怎么用"，或代码中新增 Bedrock import 时自动触发
-   对应文件：`.codemaker/skills/bedrock-component-lookup/SKILL.md`

#### @projectKnowledge 项目知识库

-   作用：查阅和管理项目知识库（名词/组件约定/业务规则/开发习惯）
-   触发方式：AI 遇到不确定的项目术语时自动查阅，或说"记录到知识库"
-   对应文件：`.codemaker/skills/project-knowledge/SKILL.md`

#### @i18n 添加国际化文案

-   作用：扫描文件中的硬编码中文，生成国际化 key 和中英文翻译
-   聊天内容：
    @/.codemaker/skills/i18n.md 目标: @src/components/AssetCard/index.tsx
-   注意：新功能暂不需要接入 i18n

### 🔍 代码质量类

#### @reviewCode 代码审查（统一入口）

-   作用：代码审查与质量保障的统一工作流，整合自查、类型优化等功能
-   触发方式：每次修改代码后自动执行，或说"审查代码"、"code review"、"自查"
-   对应文件：`.codemaker/skills/review-code/SKILL.md`

#### @fixTypes TypeScript 类型优化

-   作用：消除 any 类型，复用全局类型定义，提升类型安全
-   触发方式：说"修复类型"、"优化类型"、"fix types"
-   对应文件：`.codemaker/skills/fix-types/SKILL.md`

#### @ruleSummarizer 规范整理

-   作用：将口述的编码规范、约定或行为准则，自动写入对应的 .mdc 规范文件
-   触发方式：说"归纳rule"、"补充到rules"、"写入规范"
-   对应文件：`.codemaker/skills/rule-summarizer/SKILL.md`

### 🎨 设计验收类

#### @designQA 设计稿验收

-   作用：对比实现页面与设计稿，确保 UI 还原度
-   触发方式：说"验收设计稿"、"对比设计稿"、"检查还原度"
-   对应文件：`.codemaker/skills/design-qa-workflow/SKILL.md`

## 三、推荐使用流程

```
1. 新窗口开始   → @useRule（加载规范）
2. 做需求       → 说"做需求" + 提供文档链接 → 自动触发需求分析工作流
3. 开发新功能   → @createPage / @createComponent（生成骨架）
4. 需要接口     → @updateApi（拉取最新接口）+ @createRequestHook（封装请求）
5. 需要全局状态 → @createModel（状态管理）
6. 需要复用逻辑 → @createHook（封装 Hook）
7. 重构优化     → @createPresenter（逻辑分离）
8. 质量检查     → @reviewCode / @fixTypes
9. 设计验收     → @designQA（对比设计稿）
10. 提交前      → @pref（最终清理和优化）
```
